# CWAT Digital Portal — Setup & Change Notes

## 1. One-time Supabase setup

1. In your Supabase project, open **SQL Editor** and run `schema.sql` (included here) once.
   This creates the `patients` / `assessments` tables, a **sequence-based** function that
   generates `CWAT_0001`, `CWAT_0002`, ... IDs atomically (so two doctors saving at the same
   instant can never collide), and **Row Level Security** so each logged-in user only ever
   sees their own patients and assessments.
2. In **Storage**, the script also creates a public bucket called `wound-photos`. If your
   project blocks `insert into storage.buckets` from the SQL editor, just create the bucket
   manually (Storage → New bucket → name `wound-photos` → Public: ON) and re-run only the two
   `create policy ... on storage.objects` statements at the bottom of `schema.sql`.
3. Replace `index.html` in your GitHub repo with the one here. The `SUPABASE_URL` /
   `SUPABASE_KEY` constants near the top of the `<script>` block are unchanged from your
   original file — update them if your project keys ever rotate.

## 2. What was fixed / added

- **Diagnosis logic bug fixed.** The old code checked "tridosha" before checking the
  top-two gap, in the wrong order, so `Vata 8 / Pitta 6 / Kapha 5` incorrectly came out as
  "Vata Dominant" instead of Tridoshaja. New logic, matching your worked examples exactly:
  - Sort the three totals descending.
  - If (highest − 2nd) **> 2** → single dominant dosha.
  - Else if (2nd − 3rd) **> 2** → mixed dosha of the top two only (e.g. Vata-Pittaja).
  - Else → Tridoshaja (all three close together).
- **Sequential patient IDs (`CWAT_0001`, `CWAT_0002`, ...)** generated by a Postgres sequence
  in the database itself, not by the browser — this is what makes it collision-proof. The
  old code generated a random `CWAT-25-A1B2`-style ID client-side.
- **Per-user data isolation.** Added `user_id` to both tables and RLS policies so each
  physician's login only shows *their own* patients in search and in the sidebar — required
  for the "one user can access his previously filled forms" requirement.
- **Sidebar restructured**: now grouped by patient first (name + ID + visit count). Clicking
  a patient expands the list of assessment dates for that patient; clicking a date opens
  that specific read-only record — matching the two-level click flow you described.
- **New Section A fields** added to match the paper form: Place of Assessment (dropdown),
  Wound Site, Duration, Recurrence, Infection Status. These are per-visit fields (stored on
  the assessment, not the patient), since they can change between follow-ups.
- **Read-only view now renders the uploaded photo as an actual image**, not just a text link.
- **Print layout tightened for a single A4 page**: smaller print-only font sizes, tighter
  spacing, `@page { size: A4; margin: 10mm; }`, and the photo is capped at a print-safe height.
- **Removed fragile inline `onclick="...('${var}')"` handlers** for search results and the
  patient directory (a patient name containing an apostrophe would previously have broken
  the page). Replaced with `data-*` attributes and event delegation.
- Basic error handling added throughout (login errors shown in the form itself, save errors
  no longer silently swallowed).

## 3. Still worth doing before real clinical use

- Turn on email confirmation (or set up an invite-only sign-up flow) in Supabase Auth
  settings so random people can't self-register as a "user."
- Consider a soft-delete / edit-audit trail if assessments will ever need correction.
- Test the print output in your actual browser/printer combination — print CSS varies
  slightly between Chrome, Edge, and Firefox.
